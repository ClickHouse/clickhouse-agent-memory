Summary

One ClickHouse cluster hosts three memory tiers. HOT (Memory engine, sub-5 ms) holds live streams and per-investigation workspaces. WARM (MergeTree + HNSW) holds persistent, vector-searchable knowledge. GRAPH (SQL self-JOINs on MergeTree edge tables) exposes relationships over the same tables. Two entry points consume this cluster: the CLI cookbook runner and the new LibreChat UI via an MCP server.

HOT -- Memory engine
WARM -- MergeTree + vector
GRAPH -- SQL JOINs on MergeTree edges

GAP Semantic / conversation-memory gap analysis

Where the system is today vs what "semantic memory for past conversations" requires.

What we already have

CapabilityTableEmbeddingStatus
Semantic search over past incidents (obs)obs_historical_incidentsyes, 768-dWARM implemented
Semantic search over past network eventstelco_network_eventsyes, 768-dWARM implemented
Semantic search over past security incidentssec_historical_incidentsyes, 768-dWARM implemented
Semantic lookup of threat intel (IoCs / TTPs)sec_threat_intelyes, 768-dWARM implemented

These are domain artifact semantic memories -- past incidents and intel. They cover "has this problem happened before" well.

What is missing

CapabilityIntended homeStatus
Recall of prior agent-user conversation turns ("what did this user ask me last week?")agent_memory_long (planned)MISSING
Distilled user preferences / decisions ("the user prefers Gemini", "we agreed to SLO 99.9")agent_memory_long with memory_type='semantic'MISSING
Session working memory for the live chat (last N turns, scratchpad)agent_memory_hot (planned)MISSING
Generic knowledge-base articles outside domain tablesknowledge_base (planned)MISSING

Doc vs implementation drift

docs/ARCHITECTURE.md defines memory_hot, memory_long (episodic / procedural / semantic), and knowledge_base -- but the implemented schema (cookbooks/shared/schema/01_schema.sql) only creates the domain tables. LibreChat today stores its own chat history in MongoDB; nothing is vectorised for cross-session recall.

PROPOSAL Agent conversation memory layer

Close the gap with two tables mirroring the existing HOT + WARM pattern, and three new MCP tools that slot into the same tier-annotated envelope. This adds "agent remembers the user" without disturbing any domain logic.

Table shapes

-- HOT: rolling live conversation (volatile)
CREATE TABLE enterprise_memory.agent_memory_hot
(
    session_id    String,
    turn_id       UInt32,
    role          LowCardinality(String),    -- 'user' | 'assistant' | 'tool'
    content       String,
    tool_name     LowCardinality(String),
    metadata      String,
    ts            DateTime64(3) DEFAULT now64()
) ENGINE = Memory;

-- WARM: persistent, semantically searchable conversation memory
CREATE TABLE enterprise_memory.agent_memory_long
(
    memory_id         UUID DEFAULT generateUUIDv4(),
    user_id           String,
    agent_id          LowCardinality(String),
    session_id        String,
    turn_id           UInt32,
    role              LowCardinality(String),
    content           String,
    content_embedding Array(Float32),
    memory_type       LowCardinality(String),   -- 'episodic' | 'procedural' | 'semantic'
    importance        Float32 DEFAULT 0.5,
    ts                DateTime64(3),
    INDEX embedding_idx content_embedding TYPE hnsw(metric='cosine')
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(ts)
ORDER BY (user_id, ts, session_id);

New MCP tools (to add under cookbooks/mcp_server/)

ToolTierPurpose
memory_conversation_window HOT Return the last N turns for the current session_id straight from agent_memory_hot.
memory_conversation_recall WARM Embed the query, rank agent_memory_long rows for this user_id by cosineDistance. Answers "what did this user ask about payments last month?"
memory_conversation_remember WARM Distilled write: agent calls this to persist a fact (e.g. "user prefers Gemini", "runbook step 3 worked"). Row lands in agent_memory_long with memory_type='semantic' and a high importance score.

Lifecycle

  1. Every chat turn is inserted into agent_memory_hot (sub-ms write).
  2. A background flush (every 5-15 min, or on session close) embeds new turns and inserts into agent_memory_long.
  3. Daily TTL moves rows older than 90 days to cold storage.
  4. The agent calls memory_conversation_recall at the start of a session to seed context with relevant past exchanges.

Recommendation

Add this as a focused follow-up PR: two DDL files, one seeder for demo turns, three new MCP tools, and a short system-prompt update for each preset so the agent knows when to recall and when to remember. No changes required to the existing five domain tools.
